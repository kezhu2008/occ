# W6 — Backend Lift (Path A, single-tenant unattended sync)

Goal: Get *your* backend off your phone. IBKR Flex + price sync runs on AWS on a schedule and on-demand, with no phone open; the iOS app becomes a thin viewer of server-computed state. **A lift, not a rebuild** (grounded in `backend-migration-review.md`): keep `LedgerCore` verbatim, move storage/secrets/schedule/notify/transport to AWS behind the existing named seams, put a thin read API in front. Single-tenant only — no auth/identity beyond a device-pinned key; multi-tenant SaaS is the conditional Path B (W7).

Success criteria (observable, testable):
- **Unattended sync works with the app closed.** With the iPhone powered off / app force-quit, an EventBridge cron invocation of the `SyncWorker` Lambda pulls the IBKR Flex statement (SendRequest → poll → GetStatement, `ndcdyn`→`gdcdyn` failover, all in-core) and the Sydney-midnight quote snapshot, appends any new facts to the server store, and re-folds projections — verified by querying the read API the next morning and seeing facts/positions that the device never touched (unattended-scheduled-sync, A8 [BL]). This is the literal pilot pain removed: no "Sync now" button machine, no "phone must be open."
- **Tax numbers byte-identical to the on-device build.** A server-side replay of the committed golden `.flex-local` bytes produces a tax projection whose `factDigest` and every CGT / franking / FX / FITO / Div-775 / option cell matches the on-device build to the cent — gated by the *same* hand-derived oracle (IND × SMSF, FY25 × FY26) that gated W1 (server-tax-byte-identical, A2 [BL]). The on-device byte-identical-replay property (`domain-core.md` A3) holds across the seam, preserved by A1 [BL]: drop all projections, replay server-side ⇒ byte-identical output. A determinism-canary Lambda replays the golden bytes on a schedule and alarms on any digest drift.
- **Pre-refactors land with zero behaviour change (R23-locked).** A `FactLogStore` protocol seam, a consolidated `ConnectionState`, and the closed D83 CGT-fallback hazard (safe-entity-fallback, R11 / A7 [BL]) all ship in-app first, with the full byte-identical-replay determinism suite (`domain-core.md` A3, preserved through the seam by A1 [BL]) and the tax-oracle suite staying green — proven by no change to any tax cell or fact digest on the existing fixtures and real `.flex-local`.
- **Idempotent server ingest.** Re-pulling the same Flex statement ⇒ the conditional put reports 0 newly-persisted facts and the content digest is unchanged; a second pull on the same Sydney day leaves the daily snapshot frozen (A28, driven server-side by A8 [BL]).
- **Secrets never on-device, never leaked.** The IBKR `{token, queryId}` lives only in AWS Secrets Manager (one secret per connection), read only to build the Flex URL — never logged, never returned in a response, never in `API`'s IAM role.

Use cases covered:
- **UC-CONN-1 — Connect IBKR to the server (one-time, founder).** The single-tenant credential is provisioned into Secrets Manager (`ledger/<tenant>/<conn>`) behind `SecretsManagerTokenStore: TokenStore`; the thin client confirms "Connected · server will sync on schedule." Supersedes the device Keychain single-item upsert (D48) — the seam supports multiple connections even though Path A uses one.
- **UC-SCHED-1 — Unattended scheduled sync (app closed).** EventBridge cron (hourly on trading days + a guaranteed pull just after 00:00 Australia/Sydney) invokes `SyncWorker` → `runRefresh`; new facts append, projections re-fold, the daily snapshot freezes — with no client involvement. The client later *observes* "last sync / next sync / status," it does not drive it (the §2.4 sync FSM inverts into server-side job locking).
- **UC-SCHED-2 — On-demand sync from the thin client.** An EventBridge rule / API route invokes the *same* `SyncWorker` handler with `{tenant, connectionId, reason:"on-demand"}`; the per-connection `jobLock` prevents a scheduled + on-demand double-pull, and the `lastPullAt` throttle respects the IBKR per-token rate limit (error 1018). This replaces UC-SYNC-5 and UC-SYNC-6's device `resyncNow()`.
- **UC-HEALTH-2 — View server sync health.** The read API surfaces last-pull-at, next-scheduled-pull, newest/oldest fact dates, fact count + digest, and `SyncHealth {active | degraded | error}` — the read-only replacement for the on-device `ResyncControlVS` / LEDGER STATE panel (UC-SYNC-7).
