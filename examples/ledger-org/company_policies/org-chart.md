## Rules

- **No empty seats.** Every row above maps to a real on-disk persona Ledger actually needs across W1–W6. **Architect and Tech-Lead stay distinct** — correctness (`architecture.md`, invariants the system enforces) vs delivery (`system-design.md`, the AWS topology/API/migration that gets operated). The real Ledger docs keep these as two different files; never collapse them. The seam *between* them — e.g. the W6 `FactStore` protocol the architect designs and the tech-lead operates against DynamoDB — is a consultation edge, not a merge.
- **`specs/` is co-owned.** `ledger-product-manager` holds the demand side (every use-case is backed by an `R#`/`A#`); `ledger-architect` holds the behavioural contract (`R#`/`A#` + invariants: R8 display/cash/quotes never move tax, A5 FactID dedup, A6 byte-identical replay, A7 cost-basis-fill idempotence, D7 Decimal-only money, D8 determinism). Both reviewers gate it — `ledger-pm-reviewer` checks the demand side, `ledger-architect-reviewer` checks the contract.
- **`system-design.md` is the tech-lead's**, separate from the architect's `architecture.md`. It owns the W6 deploy topology (Lambda running `LedgerCore` verbatim, EventBridge scheduled+triggered sync, DynamoDB multi-tenant fact store, Secrets Manager tokens, SNS→APNs push, thin read API + Cognito) and the Path A (single-tenant unattended) → Path B (multi-tenant SaaS) migration plan. `architecture.md` is the invariants the system enforces today; `system-design.md` is the next-phase topology that has to be run.
- **Every producing role MUST have a reviewer.** The CEO's reviewer is the internal `ledger-board-reviewer` (then founder sign-off — this is the answer to "who reviews the CEO"). **Reviewers and HR are the only roles without their own reviewer.** QA is the one exception in the other direction: it produces tests, so those tests ARE reviewed — by its dedicated test-reviewer (`ledger-qa-engineer-reviewer`), for non-vacuousness via mutation probe.
- **`ledger-qa-engineer` is product QA — the second level of QA**, and it stays in the chart. The two levels are different jobs: the **reviewer personas** (`board`/`pm`/`ux`/`architect`/`tech-lead`/code reviewers) are **agentic-integrity QA** — they catch AI-induced failure (hallucination, fabricated "done", vacuous tests, spec drift) per-artifact. The **QA engineer** is **product QA** — does the actual software work for the user, via holistic cross-task code review + acceptance/integration tests against the spec `A#` at the milestone Integrate phase. Reviewers protect the process; QA protects the product. Keep both.
- **Stack adapters** tell the tech-lead which `superpowers` stack-skills to pull into each executor's chain: `swift` for `LedgerCore` and the W6 server lift; `ios, swiftui, swift` for the app. Managers and reviewers of non-code artifacts carry none.
- **HR is special — not in the reporting chain.** It runs between waves, reading MEMORY + `performance/` and promoting to the right tier; it gardens the shared `knowledge/` tier (GC, supersede, budget). HR = incremental within-role; the CEO's strategic review = structural cross-role. The current codification queue HR owns includes D75 oracle-independence, D58 serial mutation probes, D63/D77 UX-gate-first, and the D85 copy-truth-guard lesson.

## The four flows (every persona's job description names all four)

- **Reporting** travels up: executor → `ledger-tech-lead` → `ledger-ceo` → founder.
- **Review** gates every artifact (a different persona, PASS/FAIL) — the producer/artifact/reviewer rows are the table above; see `handoff-and-review.md`.
- **Consultation** resolves ambiguity laterally/upward without guessing — e.g. `ledger-tech-lead` → `ledger-architect` "this W6 decomposition is blocked by R8; the `FactStore` protocol must not let a server store move a tax cell" — see `consultation.md`.
- **Escalation** reaches the founder only for direction-changing calls — e.g. W6 Path A vs A-then-B, the licensed-quote-feed decision, monetisation — see `automation.md` / `decision-escalation.md`.
