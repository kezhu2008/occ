# W2-T2 — Multi-month statement stitching (ordered, de-duplicated, idempotent, gap-checked merge into the append-only fact log)

Goal: A single context-sized unit — given several capped-date-range Flex statements pulled across a long history window, produce ONE merged, ordered, de-duplicated `[RawRecord]` (and the facts it reconciles to) such that the merged result is **byte-identical** to ingesting the equivalent single full-history statement, is **idempotent** under re-pull/overlap, and **detects coverage gaps** between adjacent windows instead of silently producing a wrong CGT. This is the `HistoryStitch` layer that sits between the `FlexWebServiceRawRecordSource` pulls and `store.ingest(...)`.

Owner: ledger-tech-lead (task author) · Executor: ledger-backend-engineer · Stack: swift (LedgerCore Foundation-only + the `App/LedgerApp/App` spine, NO UIKit/SwiftUI/SwiftData)

---

## Why (grounding — `waves.md` W2, `decisions.md` constraints)

IBKR Flex **activity queries cap their date range** (`waves.md` Known constraints: "Flex activity queries cap their date range — full history needs several stitched pulls"). The real Acme Company account (`U0000000`) has history older than any single Flex window returns, so W2's `FlexWebServiceRawRecordSource` (SendRequest → poll reference code → GetStatement, `Ingest/FlexWebServiceRawRecordSource.swift` + `FlexTransport.swift`) must be driven over **multiple overlapping windows** and the results stitched. W1 already proved single-source idempotency via `FactID = hash(stableSourceID, kind)` (A4, idempotent double-import) and byte-stable canonical JSONL (`Facts/FactStore.swift`, `(eventDate, stableSourceID)` order, sorted keys, exact `Decimal`). This task makes the **multi-window** path correct and is the prerequisite for the W2 done-when "numbers reconcile vs a manual file import" (`waves.md` W2 Reconciliation gate).

The seam already exists in the spine: `App/LedgerApp/App/HistoryStitch.swift` + `RecordingFlexTransport.swift` (architecture.md §App spine). This task specifies its behaviour and proves it.

## Spec (the reviewer checks against THIS)

Build / harden `HistoryStitch` as a **pure, deterministic** function and wire it into the real-ingest path. No `Date()` / `TimeZone` / `Locale.current` / `UUID()` inside the fold (D8/I4) — the set of windows and "now" are explicit inputs.

### S1 — Window planning (capped ranges, deterministic, overlapping)
- Input: a requested `[start, end]` history span (AU calendar `DateComponents`, no time-of-day, consistent with `Fact.eventDate`) + a max window length (the Flex cap; default 365 days, an explicit input, NOT hardcoded as a literal in the fold).
- Produce an ordered `[FlexWindow]` covering `[start, end]` with a **deliberate overlap** between adjacent windows (default overlap ≥ 1 day, explicit input). Overlap is the safety margin that lets dedup absorb boundary rows; it is intentional, not a bug (mirrors the W1 "dedup absorbs the overlap" property, `backend-migration-review.md`).
- Windows are produced newest-or-oldest-first by an explicit, documented order; the order MUST NOT affect the merged output (proven by S6).

### S2 — Per-window pull + parse (reuse W1, do not re-implement)
- Each `FlexWindow` is pulled via the existing `FlexWebServiceRawRecordSource` over an injected `FlexTransport` (the fakeable HTTP seam — `RecordingFlexTransport` / a canned-bytes fake in tests, NEVER live network in CI, mirroring the W1 `QuoteTransport` discipline, architecture.md §2 and D32). Each pull yields `[RawRecord]` via the **already-built** `FlexXMLParser` — this task adds NO new parsing.
- A per-window pull failure (transport error, non-Success reference code, timeout) is surfaced as a typed per-window result, NOT a thrown abort of the whole stitch: a single failed window degrades to a **flagged coverage gap** (S4), prior windows' data survives (degrade-safe, the D45 / "leave prior data intact" discipline).

### S3 — Ordered, de-duplicated merge (the core)
- Concatenate all per-window `[RawRecord]` and reconcile through the existing `Reconciler` → `[Fact]`, then `store.ingest(...)` into the append-only `FactStore`.
- **De-duplication is on `FactID = hash(stableSourceID, kind)` only** (A4 idempotent double-import; A5 content-addressed collision-safe FactID, `Support/Ids.swift`, `Facts/FactStore.swift` `append` skips a present id). A row that appears in two overlapping windows produces the **same** `FactID` and is merged to one fact — never counted twice. Do NOT de-duplicate on raw-record object identity, on row position, or on `nativeAmount` (those are not stable across windows).
- The merged log is written in the canonical `(eventDate, stableSourceID)` total order via `CanonicalJSON` (existing `FactStore` behaviour) — the stitch MUST NOT introduce its own ordering of the fact bytes; ordering is the store's invariant and the stitch feeds it an unordered set.
- The two-fact rule is preserved: a foreign-dividend row stitched from any window still yields BOTH its `.dividend`/`.distribution` and its paired `.withholding` fact (distinct `FactID` via `kind`) — the merge never collapses them (data-model §4.1).

### S4 — Coverage-gap checking (the safety property — do not silently under-cover)
- After planning, verify the produced windows **actually tile** `[start, end]` with no UNINTENDED hole: adjacent windows must overlap or abut; a window that failed to pull (S2) or a planning hole leaves an explicit **`coverageGap(from:to:reason:)`** in the stitch result.
- A coverage gap is SURFACED (returned in the `StitchResult`, routed to the existing GAPS inbox surface as a sync-level gap, distinct from a cost-basis `transferIn` gap), NEVER silently swallowed. Rationale: an unnoticed history hole would drop acquisition parcels and **under-state a CGT gain** — the same class of correctness failure the cost-basis-gap feature guards (I9: "a gain is never under-stated").
- A fully-covered, all-windows-succeeded stitch returns `coverageGaps == []` (non-vacuous: the happy path is clean).

### S5 — Idempotent re-stitch (no cursor)
- Re-running the full stitch (same span, same windows, re-pulling the entire overlapping set) is a **byte-stable no-op** on an already-populated real store: `FactStore.digest()` (SHA-256 over canonical bytes) is unchanged, fact count is unchanged (A4 idempotent double-import; "no since-cursor — the IBKR queryId defines the window, dedup absorbs overlap", system-design.md). This is what makes scheduled / on-demand re-sync safe and is the W2 analogue of the W1 A4 double-import test.

### S6 — Determinism / order-independence (I4 / A3 replay determinism)
- The merged `[Fact]` set and the resulting `FactStore.digest()` are **independent of window order and overlap size**: stitching the same `[start,end]` with {oldest-first, 365-day windows, 1-day overlap} vs {newest-first, 180-day windows, 30-day overlap} yields a BYTE-IDENTICAL fact log. Proven by test (S6 below). No wall-clock, locale, or RNG inside the stitch.

### S7 — Wiring (real path only, demo untouched)
- Wire `HistoryStitch` into `AppContainer.ingestReal(...)` / `load()` for `source == .realStatement` (architecture.md §AppContainer). The demo/bundled-fixture path (`ingestW1`, `…/Ledger/demo`) is UNTOUCHED (D49/D85 — demo and real never share `facts.jsonl`).
- The Flex token is read from the existing Keychain `TokenStore` (D48) and is NEVER logged (D31) — the stitch passes the credential through to the transport and logs windows/counts/digests only.

## Acceptance criteria — BINARY (offline, against the real on-disk `.flex-local` bytes + a fake transport, the D38 fake-transport-over-real-bytes convention)

Tests live in `LedgerCore`/spine test targets, run with the sandbox enabled, no live network. Each assertion is RED-first (mutation-proven) and against an INDEPENDENT oracle (the single-full-history ingest), per the D45 discipline.

- **A-W2T2.1 — Stitch == single-statement (the keystone).** Split the real full-history `.flex-local` statement into ≥ 3 overlapping capped windows fed through a `RecordingFlexTransport` (canned per-window bytes). Assert the stitched `FactStore` is **BYTE-IDENTICAL** (same `digest()`, same fact count, same canonical bytes) to ingesting the equivalent single full-history statement. A mutation that drops a window or skips the merge → RED.
- **A-W2T2.2 — Overlap de-dup (A4 idempotent double-import / A5 collision-safe FactID, no double-count).** Two adjacent windows share an overlap region containing ≥ 1 trade, ≥ 1 dividend+withholding pair, and ≥ 1 option lifecycle row. Assert the merged fact count equals the de-duplicated count (the overlap rows appear once each; the dividend/withholding pair both survive as distinct `FactID`s). A mutation that de-dups on raw-record identity instead of `FactID` → RED (double-counts or drops the withholding).
- **A-W2T2.3 — Idempotent re-stitch (no cursor) (A4).** After A-W2T2.1, re-run the full stitch over the same windows: assert `digest()` and fact count are UNCHANGED (byte-stable no-op). A mutation that appends instead of dedup-skipping → RED.
- **A-W2T2.4 — Order / overlap independence (A3 replay determinism / I4).** Stitch the same span with {oldest-first, 365d, 1d-overlap} and with {newest-first, 180d, 30d-overlap}; assert the two `FactStore.digest()` values are EQUAL and equal to A-W2T2.1's. A mutation introducing a wall-clock / set-iteration-order dependence → RED.
- **A-W2T2.5 — Coverage gap surfaced (the safety net).** Plan windows that leave a deliberate hole (drop a middle window's pull / request a non-tiling plan); assert the `StitchResult` returns a non-empty `coverageGaps` with the correct `[from,to]`, the gap is routed to the GAPS surface, and prior windows' facts are intact (degrade-safe). The fully-covered happy path asserts `coverageGaps == []` (non-vacuous negative control).
- **A-W2T2.6 — Reconciliation vs manual file import (the W2 done-when).** The stitched real fact log reproduces W1's manual-file-import tax numbers: the full `TaxProjection` (EOY cells + CGT disposals, canonical JSON) over the stitched log is BYTE-IDENTICAL to the same projection over a single manual `FileRawRecordSource` import of the full statement (the `waves.md` W2 "numbers reconcile vs a manual file import" gate). A mutation that loses a stitched parcel → moves a CGT cell → RED.
- **A-W2T2.7 — Token never logged / demo untouched.** Grep-clean: the Flex token never appears in any emitted log line (D31); the `…/Ledger/demo` `facts.jsonl` is byte-unchanged by a real stitch (D49, mutation-proven isolation, mirrors `FactStoreIsolationTests`).
