> **The single most important W6 pre-refactor** (`backend-migration-review.md` §1.5/§4.2, system-design
> §3 line 109: *"This is the single most important code change enabling the lift."*). Today
> `ProjectionStore` is already a protocol (the W1 swap-point), but `LedgerStore` holds a **concrete**
> `FactStore` (`Projection/LedgerStore.swift:35`) — the durable log is the **one** core seam with no

> R8-locked, byte-identical-replay preserved (the on-device `byte-identical-replay` property,

> `DynamoFactLogStore` has nothing to conform to until this lands.

Goal: Extract a `FactLogStore` protocol from the concrete `FactStore` that `actor LedgerStore`
holds directly, bind the existing on-disk JSONL implementation to it unchanged (`FileFactLogStore`),
and route every durable-log access in the core through the protocol — so the W6 server store is a
drop-in conformer. A pure in-app refactor: **no fact changes, no projection-output changes, no tax
movement, byte-identical replay preserved.** Plus two cheap, co-located consolidations the lift wants
(the `ConnectionState` collapse and the D83-T5 CGT-fallback close), and an additive **flagged-only**
note for the IBKR-restatement revision path (deferred — built in OD4, not here).

Spec (exactly what to build — the reviewer checks against THIS):

**1. Introduce the `FactLogStore` protocol (the load-bearing change).**
- Add `FactLogStore` to `Sources/LedgerCore/Facts/` — the exact three-member contract from
  `specs/backend-lift.md` R1 and system-design §3:
  ```swift
  public protocol FactLogStore: Sendable {
      func append(_ facts: [Fact]) async throws -> Int   // # newly persisted (FactID-idempotent; A3 dedupe)
      func allFacts() async throws -> [Fact]             // canonical (eventDate, stableSourceID) order
      func digest() async throws -> String               // SHA-256 over the sorted canonical log (FactLog.digest)
  }
  ```
- The protocol is the **only** durable-log surface the rest of the core uses. It exposes **no**
  update/delete member — the log stays structurally append-only (`FactStore.swift:32-147`; spec R8/A16).
- It must remain **Foundation-only** (no UIKit/SwiftUI/SwiftData/CryptoKit) so `PortabilityLockTests`
  stays green and the protocol itself is server-liftable by construction (spec R4; D4/D30).

**2. Bind the existing on-disk store to it as `FileFactLogStore` — implementation unchanged.**
- The current `FactStore` (`Facts/FactStore.swift`) is the only concrete durable log: single
  `facts.jsonl`, one fact per line, the entire file re-sorted in `(eventDate, stableSourceID)` order
  and rewritten atomically on every append via `CanonicalJSON` (sorted keys, exact `Decimal`),
  FactID dedupe on append (the O(1) `Set<FactID>` guard, A3 idempotent-server-ingest), SHA-256 content digest via
  `FactLog.digest` over the canonical bytes, per-source directory (`…/Ledger/{demo|real}`, D49).
- Make it conform to `FactLogStore` **without changing any of that behaviour** — `append` returns the
  count of newly-persisted facts (re-import of a present FactID is a byte-stable no-op, A3 idempotent-server-ingest),
  `allFacts()` returns the canonical-ordered facts, `digest()` returns the existing SHA-256 content
  digest. Renaming the type to `FileFactLogStore` is allowed; preserving `FactStore` as the conforming
  type name is also allowed — the reviewer cares only that the concrete file impl conforms and is
  behaviourally identical. Keep the hand-vendored pure-Foundation `SHA256.swift` (no CryptoKit, D4).

**3. Route `LedgerStore` (and every other durable-log consumer) through the protocol.**
- Change `actor LedgerStore` to hold `any FactLogStore` instead of the concrete `FactStore`
  (`LedgerStore.swift:35`); `LedgerStore` remains the single serialisation point / request-handler
  boundary (`backend-migration-review.md` §1.4). Construction (the per-source store built in
  `AppContainer.makeStore` under the D49 subdirectory) injects a `FileFactLogStore` — the only change
  at the call site is that the injected type is now behind the protocol.
- Sweep every other reference to the concrete log type and re-point it at the protocol (the
  class-closing sweep the review gate requires for a seam-extraction task). The `append` /
  `allFacts` / `digest` members are the **complete** durable-log surface; if any consumer reaches a
  method not on the protocol, that is a leak the reviewer FAILs (mirrors A1's mutation: a path that
  bypasses the protocol to touch the concrete store FAILs the seam test).

**4. Consolidate the scattered connection booleans into one `ConnectionState` (co-located cleanup).**
- The App layer mirrors ~8 connection/sync booleans across `AppModel` + `SyncViewModel` (divergence
  risk; `backend-migration-review.md` §2.5) and collapses token locked-vs-absent into
  `(try? load()) ?? nil`. Replace them with one enum
  `ConnectionState { notConfigured | active | degraded | error(reason) }` (system-design §6.3 line 190;
  spec R21), `reason` token-free (D31). This is App-layer plumbing only — it touches no core fold, no
  fact, no projection output. Mechanical state-mirroring, not new behaviour: the existing
  `ResyncResult` surfaces (`AppContainer.swift:168-181`) map onto the new enum 1:1.

**5. Close the D83-T5 CGT-fallback hazard (conservative 0%, never silent 50%).**
- Today a second discovered account with no `EntityDef` silently inherits the 0.5 CGT discount
  (`RealDirectory` is hand-wired to one entity/one account; `backend-migration-review.md` §3). Change
  the entity-discount resolution so an account with **no** `EntityDef`/assignment computes its
  disposals with a **conservative 0% CGT discount** and is flagged "needs assignment" — **never** a
  silent 50% (spec R11/A7). At N=1 with the single Acme entity assigned, this is observably a
  no-op (the assigned account keeps its Company-rule discount); the change only bites an *unassigned*
  account, which does not exist in the committed fixtures — so all goldens stay byte-identical.

**6. Flag (do NOT build) the IBKR-restatement revision path.**
- Identity is `(stableSourceID, kind)` with no revision concept; if IBKR restates a row under the same
  id+kind with a corrected amount, dedupe silently drops the correction (`FactStore.swift:100`;
  `backend-migration-review.md` §1.5). Add a code-level `// deferred per backend-lift.md OD4` note + a one-paragraph
  consultation back to `ledger-architect` proposing the additive `revision: Int` / `supersedes: FactID?`
  fact-model change (system-design §5 line 135, §10 line 236). **Out of scope for this lift** per
  `specs/backend-lift.md` Out-of-scope + OD4 — flagged, version-bumping when opened, **not built here**.

**Hard constraints (the bar):**
- **R23 re-lock holds (binding; the display firewall historically called "R8").** Display/cash/daily-P/L/snapshot/overlay/quote inputs still never enter the
  fact log, the lot ledgers, or any tax/CGT/income/FXGain/EOY/optimise/Div775 input — `backend-lift.md` R23
  re-locks the `domain-core.md` R23 `display-never-moves-tax` invariant across the lift. The only
  intentional tax mover remains `Div775Engine` (D74/D84). No tax cell moves.
- **Byte-identical-replay holds (binding).** Same facts + same `ProjectionConfig` ⇒ byte-identical
  projections; drop all projections + replay ⇒ byte-identical output, through the protocol exactly as
  through the concrete store (the on-device `byte-identical-replay` property, `domain-core.md` A3,
  re-run through the seam — proven server-liftable here by `backend-lift.md` A1/A2). No
  `Date()`/`UUID()`/`TimeZone.current`/`Locale.current`/randomness introduced (D8/spec R6).
- **No scope creep.** No server store, no async-storage rewrite of the file impl beyond what the
  protocol signature requires, no `revision` field, no new projection, no behaviour change. Storage
  engine choice (Dynamo vs Aurora) is system-design's, behind this seam (OD3) — not this task.
- **Rollback = revert the branch** (clean, self-contained; no migration, no data change).

Acceptance criteria (binary; what the tests must prove — the `A#` are from `specs/backend-lift.md`):

  **only** through `FactLogStore` (a `FileFactLogStore` injected), the A2 tax-oracle suite
  (`TaxAcceptanceTests`) and the byte-identical-replay rebuildability suite (`RebuildabilityTests`,
  the `domain-core.md` A3 property re-run through the seam) pass **unchanged**, and
  the `TaxProjection` canonical digest equals the pre-refactor golden **`7f6784e1…`** by exact string
  equality. **Mutation (RED-first, D45):** a path that bypasses the protocol to touch the concrete
  store FAILs the seam test.
- **A16 — append-only + canonical-order + exportable.** The protocol exposes **only**
  `append`/`allFacts`/`digest` (no update/delete-fact operation); `allFacts()` returns canonical
  `(eventDate, stableSourceID)` order; a per-source export of the canonical bytes round-trips to the
  same `digest()`. **Mutation:** adding any mutate/delete member, or returning facts out of canonical
  order, FAILs.
- **A2 / byte-identical-replay + tax-oracle parity stays green.** The full byte-identical-replay suite
  (the `domain-core.md` A3 property, proven server-liftable by `backend-lift.md` A1/A2) and the W1 tax
  oracle (**IND × SMSF, FY25 × FY26**) stay green; every CGT/franking/FX/FITO/Div-775/option cell and the
  fact `digest` are **unchanged** on every bundled fixture (`flex_dummy.xml`, digest `7f6784e1…`) **and**

  FAILs.
- **Portability lock stays green.** `PortabilityLockTests` passes — `LedgerCore` (incl. the new
  protocol) imports **Foundation only**, no UIKit/SwiftUI/SwiftData/CryptoKit. Server-liftable by
  construction (R4).
- **R8 re-lock stays green.** `DailyPLR8RelockTests` (+ the App-layer R8 guards) pass non-vacuously —
  the negative-control leak of `snapshot.mark`/`currentFX`/`netCash` into a tax input still turns the

- **D83-T5 close is proven.** An offline test (an account with **no** `EntityDef`) shows the

- **No restatement code shipped.** The revision/supersede path is a `// deferred per backend-lift.md OD4` note + an architect
  consultation note only — grep confirms no `revision`/`supersedes` field was added to `Fact`
